# 星座帝国 - 团队协作机制

> **自动驱动工作法** - 不需要领导推动，团队自己转不停！
> 
> **版本：** v1.0  
> **生效日期：** 2026-03-15  
> **适用范围：** 星座帝国项目组

---

## 一、核心原则

### 1.1 主动驱动
```
❌ 旧方式：等领导安排任务 → 做完等验收 → 等领导问才汇报
✅ 新方式：自己拉任务 → 做完主动更新 → 定时汇报进度
```

### 1.2 透明可见
- 所有任务在看板上公开
- 所有人实时看到进度
- 完成一个更新一个

### 1.3 小步快跑
- 每天至少提交 3 次代码
- 修复一个 BUG 就测试一个
- 有问题立即标记，不阻塞

---

## 二、角色与职责

### 2.1 小策（策划 + 代理负责人）

**职责：**
- 输出策划方案和实现方案
- 跟进小科进度，协调资源
- 小科修复完 → 叫小健测试
- 测试通过 → 更新看板 → 拉取下一个任务
- 每 2 小时向老大汇报一次进度

**工作流：**
```
1. 查看 TASKS-BOARD.md，了解当前进度
2. 输出下一个任务的详细方案（如有需要）
3. 通知小科执行任务
4. 小科完成后，通知小健测试
5. 小健测试通过，更新看板状态
6. 每 2 小时向老大汇报进度
7. 遇到阻塞，标记并说明原因，跳过做下一个
```

**汇报时间：**
- 15:00 - 进度汇报
- 17:00 - 进度汇报
- 20:00 - 每日总结

---

### 2.2 小科（开发 + BUG 修复）

**职责：**
- 修复 BUGLIST 中的 BUG
- 实现策划报告中的功能
- 保证代码质量，及时提交

**工作流：**
```
1. 查看 TASKS-BOARD.md，拉取一个 P0 任务
2. 开始任务 → 更新看板状态为"进行中"
3. 修复/实现功能
4. 本地测试通过 → 提交代码（git commit）
5. 更新看板状态为"已完成"
6. 拉取下一个任务
```

**提交规范：**
```
git commit -m "类型：简要描述"

类型说明：
- fix: 修复 BUG
- feat: 新增功能
- refactor: 重构代码
- docs: 文档更新
- chore: 杂项（配置、依赖等）

示例：
git commit -m "fix: 时间水晶传送 500 错误"
git commit -m "feat: 破产保护机制"
```

**提交频率：**
- 每天至少 3 次提交
- 每个任务完成后立即提交
- 不要攒一堆再提交

---

### 2.3 小健（测试）

**职责：**
- 测试小科修复的 BUG
- 测试小科实现的功能
- 发现新 BUG 记录到 BUGLIST

**工作流：**
```
1. 接到小策测试通知
2. 立即测试对应功能
3. 记录测试结果
4. 测试通过 → 通知小策更新看板
5. 测试失败 → 记录 BUG 到 BUGLIST → 返回小科修复
```

**测试反馈格式：**
```
【测试结果】
任务：XXX
状态：✅ 通过 / ❌ 失败
问题：（如有）
截图/日志：（如有）
```

**响应时间：**
- 接到通知后 30 分钟内完成测试
- 紧急任务 10 分钟内响应

## 三、任务管理

### 3.1 任务看板（TASKS-BOARD.md）

**结构：**
```markdown
## 🔴 P0 - 本周必须完成

### 🔄 进行中
- [ ] 任务描述 - **负责人 + 状态**

### ⏳ 待开始
- [ ] 任务描述

### ✅ 已完成
- [x] 任务描述 - 已完成
```

**更新规则：**
- 开始任务 → 移到"进行中" → 标注负责人
- 完成任务 → 移到"已完成" → 标注完成时间
- 遇到阻塞 → 标注原因 → 跳过做下一个

### 3.2 优先级定义

| 优先级 | 说明 | 完成时限 |
|-------|------|---------|
| P0 | 影响核心体验，必须本周完成 | 7 天 |
| P1 | 重要但不紧急，近期完成 | 14 天 |
| P2 | 优化类任务，中期完成 | 30 天 |

### 3.3 任务拉取规则

```
1. 优先做 P0 任务
2. P0 任务做完再做 P1
3. 遇到阻塞标记原因，跳过做下一个
4. 不要同时做多个任务（专注）
```

## 四、沟通机制

### 4.1 定时汇报

**小策向老大汇报：**
- 15:00 - 进度汇报（文字）
- 17:00 - 进度汇报（文字）
- 20:00 - 每日总结（文字）

**汇报内容：**
```
【进度汇报】
时间：2026-03-15 15:00

✅ 已完成：
- 任务 1
- 任务 2

🔄 进行中：
- 任务 3（小科，预计今天完成）

⏳ 待开始：
- 任务 4

❌ 阻塞：
- 任务 5（原因：XXX，解决方案：XXX）

📊 总体进度：P0 完成 5/11（45%）
```

### 4.2 即时通知

**需要立即通知的情况：**
- 重大 BUG 修复完成
- 测试发现严重问题
- 任务完成度达到里程碑（50%、100%）
- 遇到无法解决的阻塞

**通知方式：**
- 直接发消息给老大
- 简洁说明情况 + 需要决策的事项

## 五、代码管理

### 5.1 Git 工作流

```
1. 拉取最新代码
   git pull origin main

2. 开发功能
   （修改代码）

3. 提交代码
   git add .
   git commit -m "类型：简要描述"
   git push origin main

4. 更新看板
   更新 TASKS-BOARD.md
```

### 5.2 提交频率

- 每天至少 3 次提交
- 每个任务完成后立即提交
- 不要攒一堆再提交

### 5.3 代码审查

- 小科提交的代码，小策抽查
- 重大功能改动，需要小策 review
- 发现代码问题，立即反馈

## 六、测试流程

### 6.1 测试触发

```
小科完成任务 → 提交代码 → 通知小策
小策 → 通知小健测试
小健测试 → 反馈结果
```

### 6.2 测试用例

**每个任务必须有测试用例：**
- 正常流程测试
- 边界条件测试
- 错误处理测试

**测试用例模板：**
```
【测试用例】
任务：XXX

测试步骤：
1. XXX
2. XXX
3. XXX

预期结果：
- XXX

实际结果：
- XXX

结论：
✅ 通过 / ❌ 失败
```

## 七、问题处理

### 7.1 阻塞处理

```
遇到阻塞 → 立即标记 → 说明原因 → 跳过做下一个

阻塞标记格式：
❌ 任务名（原因：XXX，解决方案：XXX，需要：XXX）
```

### 7.2 升级机制

```
阻塞超过 4 小时 → 通知老大
阻塞超过 24 小时 → 重新评估任务优先级
```
